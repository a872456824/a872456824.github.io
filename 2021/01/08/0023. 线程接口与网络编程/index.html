线程接口

线程模型

线程是怎么产生的？

  1. 内核线程
  2. 用户线程

一开始计算机没有线程，只有进程。为了能处理更多的并发，并提升效率，从而发展了线程。

进程与线程本质上并无区别，如果只创建了一个进程，那这个进程也就是主线程。

pthread_equal

pthread_create 创建的线程 ID 会放在第一个参数 pthread_t *thread 里面，不过这个 ID 不是一个数字类型，因此，在判断两个线程 ID 是不是同一个线程，不能用两个 ID 是否相等来判断，他们并不能做数学运算，得用 pthread_equal。

pthread_exit：

  • 线程自杀，自己调度，自己死。

  • 自杀之前，会先检查自己需要完成的任务（atexit里面的操作）有没有完成，如果没有，则弹出，完成任务，然后自杀退出。

  • 共享内存的信息都不会被释放，除非该线程是最后一个线程。

pthread_cancel：

  • 线程被取消，他杀，别人杀你
  • 发送信号给该线程，请求杀死该线程，至于该线程是否可以被取消，取决于该线程的状态与类型
  • pthread_setcancelstate：决定线程是否能被取消的状态值，默认是允许被取消；如果不允许被取消，该信号会进入排队状态，直到该线程允许被取消。
  • pthread_setcanceltype：决定线程被取消的类型，默认是推迟，也可以是异步。
    • 异步取消意味着线程可以随时被取消，通常是立马被取消，但系统并不保证会立马被取消。
    • 推迟意味着线程的取消会被延迟，直到遇到下一个取消点函数的调用。

exit：

  • 关闭整个进程，从而关闭所有线程。
  • 所有的标准 IO 流会被刷新并关闭，跟 return 0相仿，同时，那些通过tmpfile 创建的文件也会被移除。

pthread_join：

  • 类似wait，等待特定线程的终结；如果线程已经被终结了，该函数立即返回。目标线程必须是joinable才能被其他线程或者进程等待。
  • retval：如果不为空，则会将目标线程的内容复制一份；如果是被取消的，则会把PTHREAD_CANCELED放入其中。
  • 如果有多个线程等待着一个线程死亡，则线程死亡后由哪个线程接手是不确定的。

pthread_detach：

  • 将线程ID标记为分离的
  • 标记为分离的线程，在线程完成任务死亡时，会自动的被释放该线程的所有信息，然后归还给系统，无需等待其他线程或者父进程给它收尸。
  • 尝试分离一个已经分离的线程，是一个未定义的行为。

pthread_yield：

  • 让出线程

  • 协同式操作系统，让出cpu就是用yield，在抢占式操作系统中，这个函数不是很推荐

网络编程

socket编程基础

  • socket套接字是什么
  • 网络编程的时候，到底在干什么

需要了解TCP、IP 的五层模型，或者 OSI 的七层模型。

  • 正常写的程序都是 OSI 七层模型中最上层——应用层
  • 应用层下面是传输层，再下面就是网络层。
  • 在网络编程中，对于开发者而言，能做的只能是选择运输层的协议，选择TCP/UDP，或者对TCP/UDP的参数进行一些修改。
  • TCP：传输控制协议，面向连接的可靠控制协议
    • TCP会先保存一份数据，然后再发送数据，等到接收方确认收着了，再发第二份数据；如果没有确认（数据丢失），则会重新再发送一遍。
    • 连接：双方都应该保存一些变量，以便描述双方之间的状态
    • 连接建立：
      • 三次握手：
      • 一：A 发信息给 B
      • 二：B 收到信息，发确认收到的信息给 A
      • 三：A 收到信息，发确认收到的信息给 B
  • UDP：用户数据报协议，无连接，不可靠的数据传输协议
    • 不管接收方有没有收着，只管发送。
    • 优点：灵活，可以随时随地发信息，不用管其他
  • IP：无论是TCP/UDP，都需要依靠底层的程序（底层交通）进行数据传输，这个底层的交通的目标或者起始点就是IP（类似现实中的IP地址），尽力而为的交付服务（即不可靠）。

  • 进程将需要传输的数据交给套接字socket， 套接字socket会选择用TCP/UDP协议经过公共的传输运输服务（因特网）传递到另一个进程。
  • 如果是TCP，发送数据之前，会自行先备份一份数据，等另一个进程确认收到了，再销毁这个备份。

  • 端口号：一种看不见摸不着的计算机设备，类似于一栋大楼里面某个房间的房间号。

  • IP地址由下一层的IP层贴上的。

  • 序列号：发送的第几份数据

  • 确认应答号：希望下次发送的数据的序号是多少

  • 首部长度：单位是字（一般四个字节为一字），一般都是32位

  • ACK ：确认

  • RST ：重置连接

  • SYN：建立连接的请求，用于三次连接

  • FIN：断开连接，用于四次挥手

  • 窗口大小：抑制对方的发送频率，避免一下子接收过多信息，如果大小为0，则不会发确认收到的信息给对方

  • 校验和：检验数据可不可靠。

socket()

  • 创建套接字，只会服务当前线程，1v1的服务
    • 一个进程可以建立多个套接字
    • 父进程绑定了一个套接字，子进程继承了父进程，也可以使用父进程的套接字
  • domin ：域
    • AF_INET（IPV4）、AF_INET6（IPV6）、AF_LOCAL、AF_ROUTE
  • type：类型
    • SOCK_STREAM：字节流，TCP本质就是一个字节流。
    • SOCK_DGRAM：数据报，UDP使用
    • SOCK_PACKET、SOCK_SEQPACKET
  • protocal：协议，一般前面两个参数会决定这个参数，如果前面已经确定，这里可以用0
    • IPPRPTP_TCP、IPPRPTP_UDP、IPPRPTP_TIPC，一般为0
  • Return Value ：成功为新的文件描述符，否则为 -1；

bind()

  • 绑定IP和端口，用于网络的交接处，即内网与外网

    ​ 比如现在有个服务器或者自己写了个服务，这个服务是用来做防火墙的，该程序会先接收内网用户的数据请求，然后验证数据有没有问题，检测敏感字段等，如果没有问题，再把数据交给外网，在这个过程中，两个网卡是相互隔离且不能有任何冲突、交互的，这个时候需要绑定一个确定的IP，使得向内网IP发送数据时，不会给外网IP发送数据。而绑定了端口，则将该套接字的服务定向到某个特定的房间号。

  • sockfd：调用socket返回的文件描述符

  • addr ：指向数据结构struct sockaddr 的指针，保存地址信息（即端口和IP地址）

  • addrlen ：设置为sizeof(struct sockaddr)

  • Return Value：-1出错，0正常

与sockaddr相关的函数

  • 主机字节序 & 网络字节序
  • 主机字节序：大端、小端模式
    • 小端：低位字节排放在内存的低地址端，高位字节排放在内存的高地址端。
    • 大端：高位字节排放在内存的低地址端，低位字节排放在内存的高地址端。
  • 网络字节序：4 个字节的 32 bit 值以下面的次序传输：先是0~7 bit， 最后是 24 ~ 31 bit。
  • 网络字节序跟本地字节序（主机字节序）可能不一样，需要转换（从主机转换到网络，以及从网络转换到主机）。

listen()：

  • 监听端口的连接，把主动套接字转为被动套接字
  • sockfd ：调用socket()返回的套接字文件描述符
  • backlog：进入队列中允许的连接数目
  • Return Value ：发生错误返回-1， 否则返回0；

网络通信中，一定有主动方与被动方，也就是常说的cs框架（client 与 server），必然有一个主动的发起方与被动服务方。

套接字默认状态是一个主动套接字，通过listen转为被动，后果就是，无法再主动，只能被动的等待别人的连接。

accept()：

  • 接受连接，被动接受
  • accept() 用来从sockfd上返回一个新的连接：
  • sockfd：必须经由socket()，bind()，listen()函数处理后的socket；
  • addr：地址，保存对方的端口地址
  • addrlen：地址长度
  • Return Value ：成功返回一个新的sockfd，原先的sockfd依然可以用accept；失败返回-1；

三次握手成功后，由这个accept()建立的套接字进行1v1数据传输

返回的sockfd包含四个部分：

  • from IP
  • from Port
  • to IP
  • to Port

accept建立的套接字，如果不使用了，最好关闭该套接字。

listen套接字则无需关闭

connect()：

  • 主动连接
  • sockfd：系统调用socket()返回的套接字文件描述符；
  • addr：保存目的地端口和IP地址的数据结构struct sockaddr
  • addrlen：设置为sizeof(struct sockaddr)
  • Return Value ：成功返回0；失败返回-1；

connect()与accept()是一对，分别在客户端和服务端执行，在此期间，完成了三次握手操作

send()：

  • 发送数据
  • sockfd：想发送数据的套接字描述符
  • msg：指向想发送的数据的指针
  • len ：数据的长度
  • flags：一般置为0
  • sendto：主要用在UDP通信中
  • dest_addr：远端要通信的网络地址
  • addlen：地址长度
  • Return Value：成功发送的字节数，错误则返回-1，并设置errno

recv()：

  • 接受数据：从 sockfd文件读取信息到buf中，最多读取len的长度。
  • sockfd：读取的套接字描述符
  • buf：读取的信息的缓冲
  • len：缓冲的最大长度
  • flags：一般置为0
  • Return Value：成功返回读入缓冲的数据字节数，错误则返回-1，并设置errno

close()：

  • 用于四次挥手，关闭文件。

  • 四次挥手时，两端都需要调用close

  • 当调用close之后，接收方的recv()返回值为0时，则说明对端关闭了连接。

    四次挥手，谁先发起，谁就是四次挥手的客户端。

    2MSL为包的最大生存时间。

三次握手，四次挥手

kill() ：

  • 给一个进程发信号
  • 系统支持的 64 个信号

signal() ：

  • 信号句柄，表示要对信号做什么事，对 signum做handler这件事
  • 信号的操作，根据UNIX版本不同会有不同的变化
  • handler ：
    • SIG_IGN：忽略这个信号
    • SIG_DFL：默认操作，大部分是杀死自己
    • the address of a programmer-defined function (a "signal handler") ：自定义，会将这个操作置为默认操作，或者阻塞该信号
  • 问题：捕鼠器问题，即可能捕捉到一个信号的同时，另一个信号刚好溜走。
