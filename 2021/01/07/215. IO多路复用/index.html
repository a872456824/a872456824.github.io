IO多路复用

IO多路复用

select

  • fd_set *readfds：文件描述符类型的指针，读文件

  • fd_set *writefds：文件描述符类型的指针，写文件

  • fd_set *exceptfds：文件描述符类型的指针，异常文件

  • struct timeval *timeout：时间结构体的超时

  • select 和 pselect 允许一个程序监控多个文件描述符，等待直到一个或者多个文件描述符的状态变成ready，即准备可以进行一些IO操作，比如非阻塞的读取文件。

  • select 和 pselect除了以下三个操作之外，其他操作均想通过：

    • 时间结构体不一样
    • select没有信号掩码（sigmask）参数
    • select可以更新timeout时间结构体，用来映射当前还有多少时间剩余（只在linux支持），pselect则不会

  • 三个独立的文件描述符集合被监视。readfds文件描述符会被查看是否有些字符变成就绪，当一个文件结束EOF时，就会变成ready；writefds文件描述符会被查看是否有空间可以写； exceptfds查看文件是否异常

  • 退出的时候，每一个文件描述符将被原地修改，表明哪一个文件描述符已经改变了状态 ，所以如果使用一个循环来调用select时，每次都需要进行初始化。

  • 如果没有任何一个文件描述符需要被监控，可以把所有的文件描述符指定为NULL

  • 四个宏定义用来处理三个集合。

    • FD_CLR：从一个集合中删除一个文件描述符
    • FD_ISSET：从一个集合中添加一个文件描述符
    • FD_SET：测试一个文件描述符是否是这个集合的一部分
    • FD_ZERO：清空集合

  • 参数：

    • int nfds：文件描述符的数量，文件描述符最大的值+1

    • fd_set *readfds：

    • fd_set *writefds：

    • fd_set *exceptfds：

    • struct timeval *timeout：间隔时间，select监控等待的最长时间

      • 如果所有的timeval结构体为0，则select会立即返回
      • 如果timeout为NULL，则select将会无期限的阻塞下去

    • sigmask：信号掩码

      程序收到信号，有三种操作：

      1. 忽略
      2. 自杀
      3. 执行特定任务

      信号掩码就是选择性的忽略一些信息，底层感知不到这些信息。

  • 返回值：

    • 成功则返回三个文件描述符集合中，文件描述符的总数。
    • 如果失败返回-1，并设置errno，同时该文件描述符不会被修改，timeout变成未定义

poll

  • 同步IO多路复用，实现IO的多路传输

  • 本质：一个线程或者处理器，能够在很短的时间内响应很多事件

    很短的时间内响应很多事件大多数是多线程、多进程的任务，但是在一个线程的情况下，也可以模拟出来。

  • 检查文件描述符的集合里面有哪些是已经ready准备进行IO操作的

    指针和数组有什么区别？

    数组如果作为一个函数的参数，会退化成一个指针；如果想把整个数组传递给函数，必须传递加上数组的大小。

  • events与revents的子网掩码event bitmasks有以下操作可以当作掩码：

    • PLLERR：设备或套接字发生异常的情况。只在revents使用与展示
    • POLLHUP：设备或者套接字被断开连接。与POLLOUT互斥，不可能一起在revents出现
    • POLLIN：数据优先读取，非阻塞的读取
    • POLLNVAL：文件描述符未打开，对这个文件做操作的话，是一个非法的val
    • POLLOUT：正常的数据可以无阻塞的出去
    • POLLPRI：读取紧急数据
    • POLLRDBAND：无阻塞的读取高优先级数据

  • 返回值：

    • 返回ready的文件描述符数量
    • -1：出现错误
    • 0：超时

epoll（面试重点）

  • 与poll类似，检查文件描述符的集合里面有哪些是已经ready准备进行IO操作的

  • 下面的系统调用被用来提供创建和管理epoll实例：

    • epoll_create：创建一个新的epoll实例，并返回一个文件描述符，该文件描述符关联到创建的新实例

    • epoll_ctl：感兴趣的事件将通过它来注册到创建的实例中

      epoll实例本质上是一个文件。

    • epoll_wait： 等待一个IO事件变成ready，如果没有则是会处于阻塞状态

  • Level-triggered 、edge-triggered：水平模式与边缘模式

    • epoll事件分发的接口可以在边缘模式与水平模式进行分发，两者的区别如下（假设设想的情况发生）：

      • 管道读取数据的一段被注册到epoll实例中
      • 管道的另一端会把2kb的数据写到管道中
      • 一个epoll_wait完成，同时将会返回管道读数据端（rfd）作为一个ready文件描述符
      • 管道的读端会从rfd中读走1kb文件
      • 一个epoll_wait完成

    • 边缘模式：类似向水中投石子。向管道中写入2kb文件，数据到了读端，触发了一次事件，管道读端会读取1kb文件，不会完全读完，而这剩余的1kb文件不会重新从管道写端到达读端，所以，这1kb文件就已经没法读取出来，只能一直阻塞着

    • 水平模式：不管来了几波数据，只要管道里有数据就读取，直到没有数据

      水平模式比较可靠，但是边缘模式也有用武之地，因为有些时候不想对某些行为响应两次，比如，按照正常逻辑，一次读取就应该把数据都读完，如果没读完，说明该数据有问题，所以无需响应两次。所以，在高并发、响应时间小的情况下，必须使用et边缘模式。如果是为了保证数据的可靠性，可以使用LT模式

epoll_create

  • 创建一个epoll实例
  • 返回一个文件描述符，所有的后续操作都可以使用
  • 如果返回的文件描述符不需使用了，应该使用close关闭，避免对系统造成负担，系统会在关闭后销毁释放相关的资源

epoll_ctl

  • 对一个epoll文件的控制
  • 合法的op参数：
    • EPOLL_CTL_ADD：注册目标文件符到epoll实例，并关联epoll文件与实例
    • EPOLL_CTL_MOD： 修改对文件符的关注重点
    • EPOLL_CTL_DEL：从epoll实例中删除目标文件符，此时的event被忽略，或者直接写NULL

epoll_wait

  • 有加个p的，说明有个信号掩码，其他都一样
